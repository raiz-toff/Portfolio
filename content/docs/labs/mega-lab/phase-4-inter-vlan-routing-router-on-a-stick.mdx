**Concepts:** Enable communication between VLANs using a router or Layer 3 switch. We’ll use R1 as a router-on-a-stick: one physical link configured as an 802.1Q trunk carrying multiple VLANs. On R1, create subinterfaces for each VLAN (e.g. Gig0/0.10 for VLAN10, .20 for VLAN20) and assign IP gateways.

**Tasks:**

- **Rewire for Router:** Connect R1’s Gig0/0 to SW1 Fa0/23 (or any unused port). On SW1, configure Fa0/23 as a trunk (`switchport mode trunk`) allowing VLANs 10 and 20.
    
- **Router Subinterfaces:** On R1:
    
    ```
    R1(config)#interface GigabitEthernet0/0.10
    R1(config-subif)#encapsulation dot1Q 10
    R1(config-subif)#ip address 192.168.10.1 255.255.255.0
    R1(config-subif)#exit
    R1(config)#interface GigabitEthernet0/0.20
    R1(config-subif)#encapsulation dot1Q 20
    R1(config-subif)#ip address 192.168.20.1 255.255.255.0
    ```
    
    No IP on G0/0 main interface. Ensure SW1’s Fa0/23 uses the same native VLAN (if set to 99, configure R1’s subinterfaces with `native` accordingly).
    
- **Update PC Gateways:** On PCs in VLAN10 set gateway=192.168.10.1; VLAN20 PCs use 192.168.20.1.
    
- **Test Connectivity:** Ping PC0→PC1 (they were on different VLANs) now should succeed via R1. Use `show ip route` on R1 to see routes to 192.168.10.0/24 and 192.168.20.0/24.
    
- **Verify ARP/Router:** On R1, `show arp` should list the PCs’ MACs. On PCs, `arp -a` should list R1’s subinterface MAC for their VLAN.
    
- **Troubleshooting Missions:**
    
    - If ping fails, check `show interfaces trunk` to ensure the correct VLANs are allowed.
        
    - Verify each PC’s IP and gateway.
        
    - Example fix: forgot `encapsulation dot1Q` on one subinterface; enable it and retest.
        

**Review:** This simulates a multi-VLAN office. You now have inter-VLAN routing configured. Remember: only the router (or L3 switch) can forward between VLANs. Save “Phase4.pkt”.
